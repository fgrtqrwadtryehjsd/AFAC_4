# 从31分到90分：深度诊断与架构重设计

## 第一部分：赛题理解

### 1.1 "动态记忆压缩"真正想考察什么？

赛题名："金融长文本Agent的**动态记忆压缩**与高效问答"

三个关键词：
- **动态**：记忆不是预构建的静态摘要，而是根据问题动态调整的
- **记忆**：不是chunk级别的检索，而是文档级别的知识建模
- **压缩**：在保留关键信息的前提下降低Token消耗

核心洞察：**赛题考的不是"怎么检索"，而是"怎么把600K的金融文档压缩成Agent能理解并精确回答的记忆"**。

### 1.2 为什么冠军能达到90分？

反推90分的参数：
- FinalScore = 100 × Accuracy × (0.7 + 0.3 × TokenScore)
- 假设TokenScore=0.7 → Accuracy = 90/91 ≈ 99%
- 假设TokenScore=0.5 → Accuracy = 90/85 ≈ 106%（不可能）
- **冠军方案的准确率接近95-99%，TokenScore在0.5-0.8之间**

这意味着：冠军方案在100题中只错1-5题。而我们错了约65题。

### 1.3 当前路线是否偏离了赛题目标？

**是的，当前路线严重偏离。**

| 维度 | 当前方案 | 赛题目标 |
|------|----------|----------|
| 记忆粒度 | Chunk级(2K chars) | 文档级(完整知识) |
| 压缩策略 | 截断(丢弃96%内容) | 信息保留式压缩 |
| 检索方式 | 关键词+向量匹配 | 问题驱动的知识定位 |
| 推理方式 | 单次CoT | 多步验证推理 |
| 记忆构建 | 无/仅索引Card | 完整文档记忆 |

**根本问题：我们用4%的文档内容试图回答需要100%内容才能确定的问题。**

## 第二部分：架构诊断

### 2.1 各维度分析

| 维度 | 当前状态 | 瓶颈严重度 | 分析 |
|------|----------|------------|------|
| 检索 | BM25+Embedding+Keyword三路 | ⭐⭐ 中 | V7.1→V9仅+0.85，检索非主因 |
| 证据融合 | RRF融合 | ⭐ 低 | 融合策略已足够好 |
| 多跳推理 | 无 | ⭐⭐⭐⭐⭐ 极高 | 完全缺失！金融题需跨章节推理 |
| 长文本推理 | 截断到25K | ⭐⭐⭐⭐⭐ 极高 | 600K文档只看4%，丢失96%信息 |
| 文档结构理解 | 仅Card索引 | ⭐⭐⭐⭐ 高 | 知道条款号但不知道条款间关系 |
| Memory机制 | Card缓存(索引级) | ⭐⭐⭐⭐⭐ 极高 | 无真正的记忆，只有检索缓存 |
| Agent机制 | 单次推理 | ⭐⭐⭐⭐ 高 | 无迭代/验证/纠正 |
| 文档级知识 | 无 | ⭐⭐⭐⭐⭐ 极高 | 不理解文档全局，无法做全局推理 |
| 题目级推理 | 逐选项CoT | ⭐⭐ 中 | 推理框架对，但证据不足 |
| 选项验证 | Self-Critique | ⭐⭐⭐ 高 | 仅做subset检查，不做事实验证 |

### 2.2 Top3瓶颈

**瓶颈1：证据不足（长文档只看4%）**
- 600K金融合同 → 仅检索25K → 96%内容未读
- 金融条款的答案可能在任何章节，不可能通过检索完全覆盖
- 一个关于"违约事件"的问题，答案可能分散在第3、7、12、25条

**瓶颈2：缺乏多步验证推理**
- 当前：一次CoT推理 → 直接输出答案
- 需要：初步答案 → 定位验证证据 → 逐选项确认/否定 → 最终答案
- 当前模型有position bias（总选A），因为没有验证机制

**瓶颈3：无文档级知识建模**
- 当前把文档当作"chunk bag"，不理解文档结构
- 金融合同有严格的逻辑结构：定义→条件→后果→例外
- 不知道"第X条"和"第Y条"之间的逻辑关系

## 第三部分：Memory Agent设计

### 3.1 核心理念：问题驱动的动态记忆

**关键原则：记忆不是预构建的摘要，而是问题驱动的动态压缩。**

```
传统RAG：  Question → Retrieval → Evidence → Answer
                     (静态，丢失96%)
                    
动态记忆：  Question → Memory Construction → Memory Retrieval → Reasoning → Answer
                    (动态，问题驱动)      (完整知识定位)     (精确推理)
```

### 3.2 Memory单元设计

**Fact单元（事实记忆）**：
```json
{
  "type": "fact",
  "source": "第三条",
  "content": "票面利率为3.5%，每年付息一次",
  "entities": ["票面利率", "3.5%", "每年付息"],
  "confidence": "exact"
}
```

**Rule单元（规则记忆）**：
```json
{
  "type": "rule",
  "source": "第十二条",
  "condition": "发行人主体评级下调至AA-以下",
  "consequence": "触发加速到期条款，持有人可要求提前兑付",
  "exceptions": ["豁免须经持有人会议审议通过"],
  "cross_refs": ["第八条", "第二十五条"]
}
```

**Numeric单元（数值记忆）**：
```json
{
  "type": "numeric",
  "source": "第六条",
  "metric": "发行规模",
  "value": "10亿元",
  "unit": "人民币",
  "context": "本期债券发行总额",
  "date": "2024年1月15日"
}
```

**Definition单元（定义记忆）**：
```json
{
  "type": "definition",
  "source": "第二条",
  "term": "违约事件",
  "definition": "包括(A)未按时支付本息;(B)交叉违约;(C)破产重整",
  "scope": "适用于本期债券全部条款"
}
```

### 3.3 Memory压缩策略

**分层记忆架构**：

```
L0: 全文（600K chars，不压缩，保留完整性）
L1: 结构压缩（~30%体积，去格式/重复/空白，零Token成本）
L2: 事实记忆（~15%体积，提取关键事实/规则/数值，Qwen Token成本）
L3: 问题记忆（~5%体积，针对特定问题的动态提取，Qwen Token成本）
```

**L1结构压缩（零Token，规则实现）**：
- 去除页眉页脚、章节编号的重复格式
- 压缩空白和换行
- 合并重复条款描述
- 保留：所有数值、所有条款内容、所有条件语句
- 效果：600K → 200K（压缩3倍，零信息损失）

**L2事实记忆（Qwen Token成本，一次构建缓存复用）**：
- 提取每个条款的核心事实
- 建立 条件→后果 映射
- 建立条款间交叉引用图
- 效果：200K → 60K（压缩10倍，极小信息损失）

**L3问题记忆（动态构建，每题不同）**：
- 根据问题，从L2记忆中筛选相关事实
- 保留与问题直接相关的所有事实+上下文
- 效果：60K → 10K（问题聚焦，零冗余）

### 3.4 避免压缩造成信息损失

**关键策略：压缩是"去除"，不是"改写"**

| 压缩类型 | 是否安全 | 说明 |
|----------|----------|------|
| 去除格式/空白 | ✅ | 零信息损失 |
| 去除重复条款 | ✅ | 保留首次出现 |
| 事实提取 | ⚠️ | 可能遗漏隐含条件 |
| 语义摘要 | ❌ | 违反规则+信息损失 |

**安全压缩的原则**：
1. 数值精确保留（3.5% → 3.5%，不能变为"约3%"）
2. 条件语句完整保留（if-then不能拆分）
3. 否定语句精确保留（"不得" ≠ "避免"）
4. 例外条款完整保留（"但...除外"是关键）
5. 交叉引用关系保留（"参见第X条"）

### 3.5 支持跨章节推理

**条款引用图**：
```
第八条(违约定义) ──引用──→ 第三条(付息条款)
                  ──引用──→ 第十二条(加速到期)
                  ──引用──→ 第二十五条(持有人会议)
                  
第十二条(加速到期) ──条件──→ 主体评级≤AA-
                   ──例外──→ 持有人会议豁免
```

当问题问到"什么情况下触发加速到期"时，Agent可以沿引用图追踪到所有相关条款。

### 3.6 支持长文档问答

**两阶段策略**：

阶段1: 全局定位（低成本）
- 用L2事实记忆定位相关条款/章节
- Token成本：~2K/问题

阶段2: 精确验证（高成本但高效）
- 只加载定位到的条款原文（而非全文档）
- 每个条款独立验证
- Token成本：~10-20K/问题

## 第四部分：Agent设计

### 4.1 架构图

```
┌─────────────────────────────────────────────────────┐
│                   Question Input                      │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌──────────────────────────────────┐
│     Analyzer Agent (Qwen)         │
│  • 问题分解                        │

│  • 生成检索策略                     │
└──────────┬───────────────────────┘
           ▼
┌──────────────────────────────────┐
│     Memory Agent (规则+向量)       │
│  • L1结构压缩 → 定位相关章节       │
│  • L2事实记忆 → 筛选相关事实       │
│  • 条款图遍历 → 扩展关联条款       │
│  • 输出: 定位结果+相关条款列表     │
└──────────┬───────────────────────┘
           ▼
┌──────────────────────────────────┐
│     Evidence Agent (Qwen)         │
│  • 加载定位到的条款原文            │
│  • 按选项分别提取证据             │
│  • 选项A: 搜索关键词→精确引用      │
│  • 选项B: 搜索关键词→精确引用      │
│  • 选项C: 搜索关键词→精确引用      │
│  • 选项D: 搜索关键词→精确引用      │
└──────────┬───────────────────────┘
           ▼
┌──────────────────────────────────┐
│    Reasoner Agent (Qwen)          │
│  • 基于精确证据推理                │
│  • 逐选项判断：✅支持/❌不支持      │
│  • 输出初步答案                    │
└──────────┬───────────────────────┘
           ▼
┌──────────────────────────────────┐
│    Verifier Agent (Qwen)          │
│  • 对初步答案中每个选项独立验证     │
│  • 定位原文精确引用                │
│  • 无明确支持的选项→删除           │
│  • 遗漏的选项→补充检索验证          │
│  • 输出最终答案                    │
└──────────────────────────────────┘
```

### 4.2 Agent协作流程

**Step 1: Analyzer Agent**
```
输入: Question + Options + Doc_IDs
输出: {
  question_type: "条款条件判断" | "数值精确比对" | "定义归属" | ...,
  search_strategy: [
    {target: "违约事件定义", sections: ["第八条"]},
    {target: "加速到期条件", sections: ["第十二条"]},
  ],
  key_terms: ["违约", "加速到期", "主体评级"],
}
```

**Step 2: Memory Agent**
```
输入: search_strategy + Doc_IDs
输出: [
  {section: "第八条", text: "违约事件是指...", clause_refs: ["第十二条"]},
  {section: "第十二条", text: "当发行人主体评级...", related: ["第八条"]},
  ...
]
```
规则实现，零Token成本。

**Step 3: Evidence Agent**
```
输入: 定位到的sections + 每个选项的关键词
输出: {
  A: {evidence: "第八条明确规定...", source: "第八条", support: "full"},
  B: {evidence: "第十二条规定...", source: "第十二条", support: "partial"},
  C: {evidence: "无直接相关条款", support: "none"},
  D: {evidence: "第二十五条例外...", source: "第二十五条", support: "contradict"},
}
```

**Step 4: Reasoner Agent**
```
输入: 所有选项的证据
输出: 初步答案 (如 "AB")
```

**Step 5: Verifier Agent**
```
输入: 初步答案 + 各选项证据
输出: 最终答案 (可能删除无支持的选项)
```

## 第五部分：冲榜策略

### 优先级排序的10个实验

#### 实验1: 选项独立验证（Option-Independent Verification）⭐⭐⭐⭐⭐
- **原理**: 对每个选项独立检索+验证，而非一次性判断所有选项
- **预期收益**: +10-20分（消除position bias，提升多选题准确率）
- **风险**: Token消耗增加2-3倍
- **Token变化**: 18K/题 → 35K/题，总约3.5M
- **适合A榜**: ✅（已知doc_id，定向检索更准）
- **适合B榜**: ✅

**实现**:
```python
for option in ["A", "B", "C", "D"]:
    # 1. 用选项文本+问题文本做定向检索
    evidence = retrieve(question, option_text, doc_ids)
    # 2. 独立验证：这个选项是否有原文支持？
    result = qwen.chat(f"根据证据，选项{option}是否有明确原文支持？")
    if result == "✅支持": selected.add(option)
```

#### 实验2: L1结构压缩（Rule-based Document Compression）⭐⭐⭐⭐⭐
- **原理**: 用规则去除文档中的冗余内容（格式、重复、空白），保留所有事实
- **预期收益**: +5-10分（短文档可以从25K证据扩展到50K+）
- **风险**: 压缩可能不完整
- **Token变化**: 证据量增加但Token消耗不增（压缩后更多信息占同样Token）
- **适合A榜**: ✅
- **适合B榜**: ✅

**效果**:
- 600K合同 → 200K压缩 → 可喂40K/题（原来只能25K）
- 30K法规 → 20K压缩 → 可喂全文

#### 实验3: 两阶段推理（Locate-then-Verify）⭐⭐⭐⭐⭐
- **原理**: 先定位相关条款，再验证每个选项
- **预期收益**: +8-15分（精确定位避免遗漏关键条款）
- **风险**: 定位可能遗漏
- **Token变化**: 20K/题 → 30K/题（多一次定位调用）
- **适合A榜**: ✅
- **适合B榜**: ✅

**实现**:
```
Stage1: "以下问题需要从文档中查找哪些条款？问题：{question}" → 定位3-5个条款号
Stage2: 加载这些条款原文 → 逐选项验证
```

#### 实验4: 全文策略扩展（Extended Full-Document Strategy）⭐⭐⭐⭐
- **原理**: 将全文阈值从50K提升到100K，让更多文档喂入完整内容
- **预期收益**: +5-8分（保险/研究/法规域大量文档可全文输入）
- **风险**: Token消耗增加
- **Token变化**: 1.84M → 2.5M
- **适合A榜**: ✅
- **适合B榜**: ⚠️（长文档仍需检索）

**文档分析**:
- 保险域: 20题，avg 125K → 约10题可全文（≤100K）
- 法规域: 20题，avg 30K → 全部可全文
- 研究域: 20题，avg 120K → 约8题可全文

#### 实验5: 条款图构建（Clause Reference Graph）⭐⭐⭐⭐
- **原理**: 构建文档内条款间的引用/依赖关系图，支持跨章节推理
- **预期收益**: +5-10分（解决跨条款推理问题）
- **风险**: 构建图需要少量Token
- **Token变化**: +0.2M（一次性构建，跨题复用）
- **适合A榜**: ✅
- **适合B榜**: ✅

#### 实验6: 动态记忆压缩（Question-Driven Document Compression）⭐⭐⭐⭐
- **原理**: 对长文档，先用Qwen提取与当前问题相关的记忆片段
- **预期收益**: +5-10分（长文档获得问题定制化的证据）
- **风险**: 压缩可能遗漏
- **Token变化**: +10K/题（一次额外调用）
- **适合A榜**: ✅
- **适合B榜**: ✅

**实现**:
```
Call1: "从以下文档片段中，提取与问题'{question}'相关的所有事实、规则和数值"
→ 输出: 精简的事实列表(约3K tokens)
Call2: 基于提取的事实列表 + 选项 → 推理答案
```

#### 实验7: 数值精确提取（Numerical Fact Extraction）⭐⭐⭐
- **原理**: 对金融报表类文档，预提取所有数值到结构化表格
- **预期收益**: +3-5分（数值比对题准确率显著提升）
- **风险**: 提取不完整
- **Token变化**: +0.1M（一次性提取，跨题复用）
- **适合A榜**: ✅
- **适合B榜**: ✅

#### 实验8: 答案一致性检查（Answer Consistency Verification）⭐⭐⭐
- **原理**: 初步答案生成后，反向验证：如果选A，原文是否明确支持A？
- **预期收益**: +3-5分（减少错误选择）
- **风险**: 可能过度删除
- **Token变化**: +5K/题
- **适合A榜**: ✅
- **适合B榜**: ✅

#### 实验9: 自适应证据量（Adaptive Evidence Sizing）⭐⭐⭐
- **原理**: 根据题型和文档长度动态调整证据量
  - 单选题: 15K, 多选题: 40K, 判断题: 10K
  - 短文档(≤100K): 全文, 长文档: 定向检索
- **预期收益**: +2-5分
- **风险**: Token可能超标
- **Token变化**: 视情况
- **适合A榜**: ✅
- **适合B榜**: ⚠️

#### 实验10: 多次检索迭代（Iterative Retrieval）⭐⭐
- **原理**: 初步推理后，如果不确定，用不同query再检索一次
- **预期收益**: +2-3分
- **风险**: Token消耗增加
- **Token变化**: +5-10K/不确定题
- **适合A榜**: ✅
- **适合B榜**: ✅
