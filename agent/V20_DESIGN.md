# V20 大重构设计 — 基于真实数据特征

## 重构动机 (痛苦的教训)

V13 = 39.12 (实际 buggy fallback 守门 +10 分, 真实能力 29%)
V13.1 / V13fix 都在 V13 上小改 → 都退分

**根因**: V13 把所有题当成 RAG 问题, 实际数据告诉我们:

### 致命发现 (2026-06-22 23:00)

题目 fin_a_001: "比亚迪 2025 营收 > 2024 营收?"
- 真实数据: 803,964 vs 777,102 (在文档前 10K 字符)
- **V13 prompt 中: 803,964 出现 0 次, 777,102 出现 0 次**
- V13 让模型回答它根本看不到的数字!

这不是检索算法的问题, 是**架构错位**: 财报的关键数据在固定位置(前 30K), V13 用 BM25/vector 散点检索, 反而漏掉了核心表格.

## 全局数据特征 (实测)

### 文档长度分布 (100 题)
- 0-50K (22 题, 全 regulatory): **V13 已全文, 模型应能答好**
- 50-100K (19 题, research/insurance 短): **V13 已全文**
- 100-200K (14 题, research/insurance 中): **V13 已全文**
- 200-500K (7 题, mixed): **V13 检索 25K — 不够**
- 500K-1M (29 题, 全 financial_reports + 部分 fc): **V13 检索 25K — 严重不够**
- >1M (9 题, 全 financial_contracts): **V13 检索 25K — 极不够**

### qwen-plus 约束
- 输入 128K, 输出 8K
- 单题预算: ~100K 输入余地

### 题型 (100 题)
- multi (65): 大头
- tf (20): 复合陈述判断
- mcq (15): 单选

### 题型实质 (核心洞察)
- **保险**: 计算题 + 条款对照 (4 doc 题多)
- **法规**: 具体条款数字精确比对 (30 工作日/6 个月/10 年/可疑交易 vs 大额交易)
- **研报**: 特定数字精确比对 (56%/9.9%/2500 亿/8894.3 亿)
- **合同**: 发行要素识别 (发行人/评级 AAA/股票代码)
- **财报**: 同公司年度对比 (营收/净利润/现金流/分红)

## V20 架构设计

### 核心理念
**不再用通用 RAG; 用"题型 × 文档类型"专属流水线**

### 分流策略

```
question → 分类 (按 domain × format) → 专属处理
```

#### 通路 1: regulatory (25 题, 全 < 50K)
**所有文档全文进 prompt**
- system: 强调"逐字精确核验, 关键差异(可疑/大额, 30/6 月, 1 年/10 年)立判错"
- 单次 LLM 调用
- V13 在此域问题不大, 改进点: 强化 tf 子陈述拆解 prompt

#### 通路 2: research (20 题, 50-200K)
**所有文档全文进 prompt** (双 doc 总长 ~150K, 接近 128K 上限, 但通常单 doc 60-100K, 双 doc 120-180K, 部分需要轻压缩)
- 关键: 研报常含表格/图表数字, 须保留这些段
- 单次 LLM 调用
- 改进: 数值精确比对 prompt

#### 通路 3: insurance (20 题, 50-200K)
**所有文档全文进 prompt**
- 但 mcq 计算题(ins_a_001/002 这类)需要让 LLM 做实际算术 — 这超出"匹配原文", 要额外计算 prompt
- 改进: 计算题专属 prompt — 强制列出每个产品的相关数值再计算

#### 通路 4: financial_reports (20 题, 200K-1M, **占退分大头**)
**关键修复**:
1. **预提取**: 离线规则从每份财报抽出"主要会计数据"章节(找到 `主要会计数据` 关键词, 取后 15K 字符)
2. **必须包含**: 营收/净利润/经营活动现金流/研发投入/股东权益/分红 等核心数字
3. 给 LLM 看: 每个 doc 的 (前 30K + 主要会计数据章节) — 约 60K 双 doc
4. 题型分流: tf 严格判断 + multi 必须每个选项给原文数字引用

#### 通路 5: financial_contracts (20 题, 200K - 1.5M)
**关键修复**:
1. **预提取**: 募集说明书的首节"发行概况"(发行人/规模/评级/期限) — 通常在前 30K
2. **预提取**: 关键条款节(转股价格/赎回/违约) — 用条款编号定位
3. 给 LLM: 双 doc 关键节合集 ~60K
4. 改进 prompt: 强化"两份文档分别核验, 不能混淆"

### 共通组件

#### A. 强化的题型分流 prompt
- **tf**: 拆子陈述, 任一无证据/有差异 → B
- **multi**: 每选项独立给出"原文精确引用"; 没引用就不选
- **mcq**: 4 个选项都要分析

#### B. 防过选硬规则
- multi 答案抽出后, 若 4 字母 (ABCD) → 必错, 减一个; 若 ≥3 字母, 在 raw 中检查每个字母对应的原文引用是否真的有
- fallback: 抽不到默认 "A" (V13 守门, 至少保住意外分)

#### C. 保留 V13 后处理 (V13 的 fallback A 是 features)
- 不改 extract_answer_from_response
- 不改 _post_process

### Token 估算

- regulatory: 25 题 × 25K = 625K (V13 也是这量级)
- research: 20 题 × 60K (双 doc 全文) = 1.2M
- insurance: 20 题 × 60K = 1.2M
- financial_reports: 20 题 × 60K (预提取片段) = 1.2M
- financial_contracts: 20 题 × 60K = 1.2M
- 总: ~5.4M ❌ 超

**优化**: 双 doc 题用 30K + 30K = 60K, 但 tf 题只需 20K, mcq 题 25K, multi 题 35K
- regulatory: 5K avg = 125K
- research: 50K avg = 1M
- insurance: 50K avg = 1M
- financial_reports: 50K avg = 1M
- financial_contracts: 50K avg = 1M
- 总: ~4.1M ⚠️ 接近上限

**或更激进**: 长文档保留"关键章节"(前 30K + 主要数据/发行概况 15K) = 45K/doc, 双 doc 90K → 接近 128K

**实际预期**: Token Score 可能从 V13 0.63 降到 0.30-0.40, 但准确率从 29% 涨到 60%+ 才划算
- Final = 100 × Acc × (0.7 + 0.3 × TS)
- 现在 V13: 0.29 × (0.7 + 0.3 × 0.63) = 25.7 (跟 V13 fix 真实分数吻合!)
- 目标 V20: 0.55 × (0.7 + 0.3 × 0.35) = 44 (?)
- 目标极高: 0.80 × (0.7 + 0.3 × 0.30) = 63

### 实现步骤

#### Phase 1 (零 token 离线)
1. 写 `agent/structured_evidence.py`: 每域专属的"提取关键章节"函数
   - `extract_annual_report_key_data(text)`: 找"主要会计数据"后 15K
   - `extract_contract_key_terms(text)`: 找"发行概况"+条款节
   - `extract_research_full_or_compress(text)`: 短文档全文, 长文档+表格段
2. 写 `agent/reasoner_v20.py`: 题型 + 域分流
3. 写 dry-run 脚本: 在 5-10 道之前 V13 错的代表性题上跑 prompt 构造 (不调 LLM), 看证据是否真覆盖关键数字

#### Phase 2 (小 LLM 验证)
4. 在 5 道关键题上调 LLM, 看回答质量
5. 离线判断: 关键数字是否被 LLM 正确识别和比较

#### Phase 3 (全量)
6. 只有 Phase 2 通过, 才跑全量

### 风险与底线

- V20 是大重构, 失败可能比 V13.1/fix 更惨
- 必须先离线 dry-run + 小 LLM 验证, 不能直接跑全量
- 失败回退: V13 原代码已恢复, 必要时可重跑 V13 原
