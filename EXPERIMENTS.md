## 赛题信息

- **赛题**：金融长文本Agent的动态记忆压缩与高效问答挑战
- **评分公式**：`FinalScore = 100 × Accuracy × (0.7 + 0.3 × TokenScore)`
- **TokenScore**：`max(0, min(1, (5,000,000 - TotalTokens) / 5,000,000))`
- **Token 预算**：5,000,000
- **模型限制**：仅 Qwen 系列 API
- **A 榜**：100 题，已知 doc_ids
- **B 榜**：100 题，未知 doc_ids
- **当前最高分**：85

1. **FinCARDS**: Query-Focused Evidence Extraction — 每题提取关键词+BM25双路召回，不再做全局文档压缩

3. **Self-RAG/Self-Critique**: 多选题只允许删除不允许添加
4. **答案多样性**: multi题从ABCD全选变为 AB/ABC/B/BD/ACD 等多样答案

### V15 — 最小安全增量（V13继承版）

#### V15 设计原则
**核心原则：推理前必须有把握；每次修改必须记录原因和预期效果。**

#### V15 run1（2026-06-22，❌失败）
| 指标 | 值 |
|------|-----|
| 总 Token | 2,712,488 |
| TokenScore | 0.4575 |
| A分布 | **A=49**（偏置回归！与V9相同） |
| 失败原因 | Stage1 Locate 调用引入 A 偏置（同 V10.1 机制：分离调用→A 偏置） |

**教训**：额外的 LLM 调用（即使是定位类）只要增加了对 A 的"预确认"，就会引入偏置。

#### V15 run2（2026-06-22，❌失败）
| 指标 | 值 |
|------|-----|
| 总 Token | 2,700,884 |
| TokenScore | 0.4598 |
| A分布 | **A=58**（比 V13 A=39 更差） |
| 失败原因 | V15 的 `_stage2_retrieve` 完全重写了证据架构（4路不同权重），破坏了 V13 的融合均衡性；题型差异化阈值(tf=120K/mcq=90K/multi=70K) 导致路径不一致 |

**教训**：不能假设增大证据量或改变权重会保持均衡性。V13 的 A=39 是特定权重+阈值组合产生的，任何改动都可能打破。

#### V15 final（2026-06-22，❌提交失败，31.17，退步至V9水平）
| 指标 | 值 |
|------|-----|
| 总 Token | 1,821,982 |
| TokenScore | 0.6356 |
| A分布 | A=46（单选），接近正常 |
| **A 榜分数** | **31.17（退步 -7.95 分！）** |

**根本原因（离线分析确认）**：
- L1 压缩的 `\d{1,3}` 规则匹配 1-999 范围内的独立数字行
- 监管法规（regulatory，全文路径 20 题）的 HTML/TXT 源文件中，条款编号 "5"、"12"、"30"、"202" 等单独成行
- 这些行被误识别为"页码"删除，导致模型丢失关键条款信息
- 全文路径 22 题中的 regulatory 20 题大量出错 → -8 分

**教训**：
- 任何文本处理规则在金融场景下都可能误伤内容（数字、条款号、年份）
- L1 压缩必须在实际文档上逐行验证，不能仅凭理论判断
- **修复方向（不值得再试）**：即使把 `\d{1,3}` 改为仅删除 PDF 生成的 "--- X ---" 格式，收益也极小，风险依然存在
- **结论：放弃所有 L1 压缩改动，V13 是当前真实最优基线**

### V16 — Qwen-Long 文件上传（2026-06-22，❌策略性失败）

| 指标 | 值 |
|------|-----|
| 日期 | 2026-06-22 |
| 策略 | qwen-long 直接上传 PDF，1000万 Token 上下文全文推理 |
| 有效答案 | 14/100（Token 耗尽，中途停止） |
| 总 Token | 5,095,891 |

**失败原因**：
- qwen-long **每次调用都把文件全量 Token 计入**（即使文件内容未变）
- 一份金融合同 ≈ 640K chars ≈ 300K tokens，14题 × 360K/题 ≈ 5M，直接耗尽全部预算
- **结论：qwen-long 不适合本赛题**（100 题 × 多份长文档，总 Token 远超 5M 预算）

**教训**：qwen-long 计费模型是"每次调用按文件 Token 计费"，只适合单次大文档分析，不适合多题目循环场景。

## V14 改动盘点（相对 V13，待 dry-run 评估）

为 2026-06-22 离线 dry-run 准备的差异清单。

| 维度 | V13 | V14 | 风险点 |
|------|-----|-----|--------|
| 全文阈值 | 单一 `50K` | 题型分桶 `tf=120K / mcq=90K / multi=70K` | 全文阈值跨过 50→120K 会让 ~20+ 题从"融合检索"切换到"全文"。需观察 prompt 是否因此变得稀释/超长，是否触发 max_tokens=4096 后还截断。 |
| 证据上限 | `mcq=20K / tf=15K / multi=25K` | `mcq=24K / tf=18K / multi=32K` | 证据池更大,Token 上升但召回更全。V13 是 1.85M total,V14 可能突破 2M 边界。 |
| 检索路径 | 3 路 RRF（clause+metric, bm25_kw, vector） | 4 路 RRF + 选项均衡（option_chunks 单独一路, RRF 权重 1.4） | V15 run2 曾因"重写检索权重打破均衡"导致 A=58。需重点检查 V14 的 4 路融合是否再次引入偏置。 |
| 选项补充证据区 | 无 | `=== 选项X补充证据 ===` 每选项独立追加（cap=2.2K/1.8K/2.6K） | V10/V11/V12 的失败原因正是"选项分离证据区→A 偏置"。V14 是融合证据池后再追加选项段,理论上不会触发,但需要肉眼读 prompt 验证。 |
| 数值定位 | 仅 card_hints.metric_contexts | 增加从选项中提取金额/百分比/日期 | 安全增量,扩大数值题召回。 |
| Self-Critique | 多选启用（仅删不增） | 关闭 | 减少一次 LLM 调用、降低波动,但理论上会减少删错选项的能力。 |
| 多选硬截 | 多选 letters>3 截到前 3 | 不截 | 风险:若模型偏向 ABCD 全选会回归 V5 问题。 |
| mcq/tf 默认 | 抽不到字母时默认 `"A"` | 返回空串 | 减少结构化"被迫选 A",但空串记为错。需观察空答案数。 |

### dry-run 重点核对项

- 4-5 道题的 V13 vs V14 prompt 文本差异(挑长文档+多选题)
- 选项补充证据区是否真有内容,还是被前面 32K 主证据吃满后被截掉
- 全文阈值切换的题：证据是否"原本检索能找到关键条款,改全文后反而稀释"
- 数值定位是否真的多出原本检不到的关键金额/百分比上下文

### dry-run 结果（2026-06-22，零 LLM 调用，零 embedding API）

样题: `ins_a_007 / reg_a_003 / fc_a_017 / res_a_001 / fin_a_012`
脚本: `dry_run_v13_v14.py`，输出: `dry_run_outputs/`

| qid | doc_chars | V13 路径/证据 | V14 路径/证据 | call_count(V13/V14) |
|-----|-----------|---------------|----------------|--------------------|
| ins_a_007 | 63K | 检索/25K | **全文/62K** | 2 / 1 |
| reg_a_003 | 20K | 全文/20K | 全文/20K | 1 / 1 |
| fc_a_017 | 1520K | 检索/25K | 检索/32K | 2 / 1 |
| res_a_001 | 64K | 检索/25K | **全文/62K** | 2 / 1 |
| fin_a_012 | 592K | 检索/25K | 检索/32K | 2 / 1 |

100 题路径切换统计:
- **路径切换的题: 10 道**（4 道 insurance + 6 道 research，doc 在 50K~120K 区间）
- 都走检索的题: 68 道（V14 evidence 上限抬 30%: 20K→24K, 25K→32K）
- 都走全文的题: 22 道（V14 几乎无变化）

**Token 影响估算**:
- V14 关闭 self-critique → 多选题平均省一次 ~5K token 调用 ≈ 100 题 60 道多选 × 5K ≈ **-300K**
- V14 evidence 上限抬 30% → 68 道检索路径 × 7K 多增 ≈ **+476K**
- V14 全文切换 10 道，单题证据 25K→60-100K → ≈ **+400K**
- 净估算 Token: V13 1.85M → V14 ~2.4M (TokenScore 0.63 → 0.52)，**TokenScore -0.11 ≈ -1.0 分**

**严重 bug 发现**:
- V14 设计的"选项补充证据"区(`=== 选项X补充证据 ===`)在所有样题里 **0 次生效**
- 原因: `_rrf_fuse_chunks_4way` 用 `text[:100]` 做去重 key，前 3 路命中后 chunk 对象已固化（`chunk_type` 为空），第 4 路 `option_chunks` 的 `chunk_type=option_X` 标签被丢弃。option_buckets 永远空。
- 设计初衷"每选项补证据缓解漏选"在当前实现下 **完全失效**
- 这意味着 V14 的"4 路融合"实际只是"3 路融合 + 给 option 路 1.4 权重加分"，对答案选项的覆盖没改善

**V14 净收益预判**:
- 正面: 10 道检索→全文切换的题可能涨准确率(原本 25K 检索可能漏关键条款)
- 正面: 关闭 self-critique 减少波动 + 省 token
- 负面: TokenScore 大幅下降 ~0.11 (-1.0 分)
- 负面: 多选题硬截 3 改成无截 → 风险 ABCD 全选回归
- 负面: mcq/tf 默认 A 改成空串 → 抽不到字母的题计 0 分(原本默认 A 起码有 25% 命中率)
- 中性: "选项补证据"实际无效

**结论: 不推荐直接跑 V14 全量**。预期净收益负面。准确率即便涨 2-3 分,TokenScore 退 1 分,整体可能持平或退步,并且暴露在两个"默认 A 移除""硬截移除"的新风险下,行为不可预测。

### 决策（2026-06-22）

- ❌ **不跑 V14 全量**——风险大于收益
- ✅ 在 V13 基础上做**单变量增量**:
  - 先试 **V14-lite**: 仅关闭 self-critique，其它保持 V13 不变（省 ~300K Token，TokenScore +0.06 ≈ +0.7 分，无准确率风险下界）
  - 或先试 **V14-fix-option-bucket**: 修 4 路融合 bug，让"选项补证据"真生效（验证设计是否真能提多选覆盖）
  - 全文阈值上调 70K-multi 这条单独验证

### 离线 dry-run 工具

每次架构改动都必须先用 `dry_run_v13_v14.py` 模板做一次 prompt 对比，**确认证据真按预期变化** 才能花预算跑全量。

## 错题归因诊断 (2026-06-22, 零比赛 Token)

抽 3 题做"召回 / 模型 / 压缩"三层归因。方法: 离线读文档定参考答案,再对比 V13 在该题的实际 prompt 内容。**目的是定位 V13 错在哪一层**,而不是为了提分。

### 题 1: `fc_a_017` (金融合同/多选, V13=ABC)

文档: text04=安克创新(965K), text05=江苏本川智能(555K) — 都是可转债募集说明书

**离线参考答案**:
- A 转股价格向下修正 → ✓ 两份都有
- B 股票代码不一致 → ✓ 300866(安克) vs 本川(代码不同)
- C 均发行 2025 年 9 月 → **✗ text04 文档里"2025 年 9 月"出现 0 次**,text05 里的"2025 年 9 月"全是审计报告期末日期
- D 均设有条件赎回 → ✓ 募集说明书标配

**预判**: **ABD** | V13 实际: **ABC** → 错(错选 C, 漏 D)

**召回情况**: V13 prompt 含 text04 段头(发行概况)+ 转股价格 40 次 + 股票代码 5 次 + 赎回 11 次 → **关键词全到了**

**归因**: **不是召回失败,是模型事实核验失败** — text04 prompt 段里没"2025 年 9 月",但模型在 25K 嘈杂证据中读到 text05 的"报告期 2025 年 9 月 30 日"就误判为"发行日期 2025 年 9 月"。**漏 D 是另一个症状**:模型在多选题倾向"早停"——选了 ABC 就觉得够了。

### 题 2: `ins_a_007` (保险/多选, V13=A)

文档: doc 1=平安智盈金生(28K), doc 2=国寿增益宝(15K), doc 16=平安富鸿金生(19K) — 合计 62K

**关键发现**: doc 1 全文 28K **"保单贷款"出现 0 次**, doc 2 全文 15K **"保单贷款"出现 0 次**。只有 doc 16 有完整"保单贷款"条款(6.2 节)。

**离线参考答案**:
- A 平安智盈金生允许保单贷款,最高现金价值 80% → **✗ doc 1 根本无"保单贷款"条款**,无法成立
- B 国寿增益宝允许保单贷款,扣欠款后 80% → **✗ doc 2 根本无"保单贷款"条款**
- C 平安富鸿金生按个人养老金制度则不允许 → ✓ doc 16 原文:"若您按照个人养老金制度投保并订立本合同的,**我们不接受保单贷款申请**"
- D 平安富鸿金生无论何种投保均不允许 → ✗ doc 16 默认可贷(现金价值 80%),只是个人养老金制度时不可

**预判**: **C** | V13 实际: **A** → 完全错

**召回情况**: V13 prompt "保单贷款"出现 17 次,全部来自 doc 16(因为另两份文档根本没有这词)。**前 25K 的证据里 doc 1/2 根本没"保单贷款"相关原文**,因为原文也不存在。

**归因**: **既不是召回也不是压缩问题,是模型的"否定推理"失败** — 选项 A 说"平安智盈金生允许保单贷款,最高 80%",但 prompt 里 doc 1 完全没有任何"保单贷款"原文支持这个表述。模型仍选 A。这是 A 偏置 + 模型"看到 80% 在 prompt 里就联想到 A"(80% 实际来自 doc 16 的富鸿金生条款,被错配到 A 选项的智盈金生)。**还把选项 A 当默认答案**。

### 题 3: `reg_a_003` (法规/判断, V13=A)

文档: 015 央行数据安全管理办法(11K) + 008 受益所有人识别办法(10K) — 都全文进 prompt

**关键事实**:
- 015 第十六条原文: "**高敏感性数据项**原则上不在终端设备和移动介质中存储,确需存储的,数据处理者应当统一规范管理"
- 008 第十八条原文: "对于市场透明度较高的客户(如上市公司),金融机构应当识别留存的受益所有人身份信息至少包括姓名、性别、国籍、出生年月以及**可以用于身份识别的照片**"

**题目陷阱**: 题干用"**敏感数据项**",法规原文用"**高敏感性数据项**"——两者**未画等号**。
此外: "上市公司受益所有人身份信息" 008 明确要"留存照片",**与 015 第十六条之间是否冲突,法规未直接定义**。

**离线判断**: 因为题干表述与 015 原文"大意相近"(都说原则上不在终端存储,确需存储统一规范),tf 类题通常按"大意"判 A; 但严格语义"敏感"≠"高敏感性",可判 B。**这题真有歧义,我估值 50-60% 是 A**。

**召回情况**: V13 prompt 全文进入,关键证据(015 第十六条 + 008 第十八条)都在。

**归因**: **如果答案是 A**: 模型表现正常; **如果答案是 B**: 模型同样陷入"听起来像→对"的陷阱,没做严格语义匹配。

### 归因总结

| 题 | 召回 | 压缩 | 模型 | 主因 |
|----|------|------|------|------|
| fc_a_017 | ✓ 关键词全到 | ✓ 全文 1.5M 检索 25K 但关键段在 | ✗ 误读 + 早停 | **模型** |
| ins_a_007 | ✓ doc 16 关键条款全到 | ✓ 三份合计 62K 检索 25K 够 | ✗ A 偏置 + 错配 80% | **模型** |
| reg_a_003 | ✓ 全文 | ✓ 全文无压缩 | ? 歧义题 | **题目歧义/模型** |

**核心结论**: **当前 V13 的瓶颈不在召回,在模型推理本身**——

1. 模型有强 **A 偏置**(尤其多选题模糊时倾向 A)
2. 模型 **不做否定推理**(选项 A 描述了 doc 1 内容,但 doc 1 没该原文,模型不会否定 A)
3. 模型 **多选早停**(选了 ABC 不继续看 D)
4. 模型容易被 **跨文档关键词错配** 误导(80% 在 doc 16 里,模型当成支持 doc 1 A 选项的证据)

### 这改变了下一步方向

之前讨论"V14a 关 self-critique / V14b 修选项补证据"——这些都是**优化召回/证据架构**,但**真正瓶颈在模型推理**。

**新方向(都是低 Token 增量)**:
- **方向 X**: Prompt 强化"对每个选项,先在证据中搜索是否有该具体表述,无则直接 ❌"——直接攻击"模型不做否定推理"
- **方向 Y**: 多选题答完后,强制让模型对所有 4 个选项都给原文引用,不允许早停
- **方向 Z**: 把"证据归属"标注进 prompt——"以下是 doc 1 的内容:..."、"以下是 doc 2 的内容:..."——让模型不能把 doc 16 的 80% 当作 doc 1 的支持

**这些方向都可以用 dry-run 离线先看 prompt 是否更合理,再考虑跑全量**。

| reg_a_003 | ✓ | ✓ | ? | **题歧义** |

### 修正后的下一步方向

诊断 3 题不足以下大结论,但当前**两个独立、可定位的瓶颈**都浮出来了:

1. **召回侧 — 同义词扩展**(低成本): 当前 BM25/vector 都用题干原词查,不能解决"贷款 vs 借款"、"身故保险金 vs 死亡保险金"这类金融领域同义。可在 query 端增加同义词扩展(规则字典即可,零 Token)。
2. **模型侧 — Prompt 强化否定推理 + 多选不早停**(低 Token): 在 CoT_PROMPT 里强制"对每个选项,如果证据无明确支持,直接判 ❌"

**这两个方向都可以离线 dry-run 验证,确认 prompt/证据真的变了,才考虑跑全量。**

需要更多题诊断才能判断这两个瓶颈各占多少比例 — 但 3 题样本里 1 题召回 + 1 题模型,说明都得修。

## 扩大归因 (15 题, 2026-06-22)

### 设计

- 离线归因脚本 `attribution_15.py`,对每题:
  - 解析 V13 dump 的 prompt,按 doc_id 拆分计算每 doc 占用字符
  - 标记"严重欠采"题: 某 doc 全文占比 > 20% 但 prompt 占比 < 全文占比 × 40%
- 零 LLM、零 embedding API(缓存全命中)

### 结果

**15 题中 3 题严重欠采**(20% 比例):

| qid | 失败 doc | 全文占比 | V13 prompt 占比 |
|---|---|---|---|
| ins_a_007 | doc 2 国寿增益宝 | 25% | 4% |
| **fin_a_012** | annual_byd_2025_report (413K) | **70%** | **0%** ← 完全消失 |
| **fin_a_009** | annual_byd_2024_report (479K) | **73%** | **0%** ← 完全消失 |

**根因**:V13 用全局 RRF 排序后挑 top-50 装到 25K 证据池,**没有 doc 级配额**。多 doc 对比题(财报对比比亚迪 vs 中国移动),某 doc 的 chunks 排名靠后就完全消失,导致那部分选项无证据可核。

## V13.1 — doc 级配额 + 同义词扩展 (2026-06-22, 未跑全量)

### 改动 (严格单变量)

1. **同义词扩展** (`agent/synonym_expander.py`): 规则字典,query 端扩展 BM25 / keyword 查询词。
   - 覆盖: 贷款↔借款、营业收入↔营收、转股价格↔转股价等 ~50 组金融领域同义
   - 否定/肯定**严格不互相扩展** (避免 V14 那种语义污染)
   - 零 Token,零外部依赖
2. **doc 级配额组装** (`agent/reasoner_v131.py` `_assemble_with_doc_quota`):
   - Phase 1: 每 doc 最低配额 = `max_evidence / n_docs × 0.6` (n=2→7.5K, n=3→5K, n=4→3.75K)
   - Phase 2: 全局填空,但每 doc 上限 = `max_evidence / n_docs × 1.6` (防单 doc 撑爆)
   - 超大 chunk 自动截断到剩余配额
3. **段按 doc_id 排序输出**: 让模型看到 "doc 1 全部 → doc 2 全部",提升按 doc 推理

**没改的**: prompt 文本、领域 system、LLM 调用次数、阈值、模型参数 — 全部保持 V13 原样。

## 🔥 重大发现 (2026-06-22 21:00) — V13 隐藏的 Extract Bug

### 发现过程

V14 prompt 实验在 8 道样题上调 LLM (¥1 验证). 表面看 V14 全部跟 V13 答案一样 (5/5 预测失败). 但**深查模型完整 raw response** 时发现:

**reg_a_010** (V13/V14 都被记为答 A, 但真值是 B):
- 模型 raw 最后明确写: `**最终答案：**\n**B. 错误**`
- 模型完成了精彩的子陈述拆解, 正确识别"大额交易"vs"可疑交易"差异
- **但 extract 抽出 A, _post_process 把 A 保留为 A**

### Bug 根因

`extract_answer_from_response()` 的策略 1 正则 `r'答案[：:]\s*([A-D]+)'` **不能跨多行匹配 markdown 格式**:
```
最终答案：

**B. 错误**
```
因为 `\s*` 不跨过 `\n\n**`.

落到策略 4 (全文 `.upper()` 抓 ABCD), 在 raw 中**最早的字母不是答案而是 doc_id 里的小写**:
```
查找文档 strict_v3_018_xxx (...)
```
`strict_v3_018` 包含小写 `c`, `.upper()` 变 `C`. `_post_process(C, "tf")` 不是 A/B → fallback **"A"**.

### Bug 影响范围

- V13 reasoner_v13.py `_post_process` 设计上 tf 抽不到字母时返回 "A" — **这是 V13 一直存在的 fallback**
- 任何模型 raw 没用"最终答案: X"严格格式, 且 raw 含 markdown / doc_id / 含小写英文字母 → **被错误识别**
- **特别影响 regulatory tf (V13 5/5 全 A) 和 research tf (V13 5/5 全 A)** — 模型可能至少有 3-5 道实际答 B, 被强转成 A

### 修复

1. `extract_answer_from_response` 新增 **策略 0**: 末尾向前找"最终答案"/"结论"关键词, 之后 200 字内找 ABCD 大写字母 (处理 markdown)
2. 策略 1 正则改为 `[\s\*]*` 允许跨行 + markdown `**`
3. 策略 2 允许去掉行首尾 `*`
4. 策略 4 改为只看末尾 1500 字内的真大写 ABCD (不 `.upper()` 否则污染)
5. 策略 5 (新增, 兜底): 仅对 tf/mcq, 全文 `.upper()` 抓 (退化为旧 bug 行为, 但放最后)
6. `_post_process` tf 抽不到字母改返回 `""` (不再默认 A)

回归测试 11/11 通过, 包括含 `strict_v3_018` doc_id 的 raw 现在能正确抽 B.
